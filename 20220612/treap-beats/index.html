Treap

關於操作 1, 2, 3, 4, 6 而言，有個好用的工具叫做 implicit merge-split treap

這邊先簡單介紹一下 treap

treap 是一種資料結構，treap = binary tree + binary heap

每個 node 會根據你的需求存取不一樣的資訊

其中 tree 的性質會根據 key 去做排序

而 heap 的性質則會利用 priorities (pri) 去做操作

注意 treap 一大特色就是它的 pri 是利用 random 的性質，讓它的時間複雜度得以維持在 O(log⁡N)O(\log N)O(logN)

treap 本身也有不同種類，一種是旋轉類型的（像紅黑數再那邊轉來轉去），甭想也知道很難刻

所以仿間比較流行的是利用 merge & split 完成的 merge-split treap

但是注意到這題的要求

我們必須時常插入以及刪除元素，但又必須區間取值，這會使我們的 index 很難維護

所以就有人發明了一種更強的 treap，就是 implicit merge-split treap

為何叫做 “implicit” 呢？這是因為我們不會直接把 index 用 “explicit” 的方式儲存在 node 中

反而是在 split 時，會根據我們的目標 index，去分成對應的兩棵 treap

透過在每個 node 儲存 size 的方式，來達成我們的目標

例如：

這會讓我們的 treap t split 成兩個小 treap：t1, t2

t1 會包含著前 10 個元素，而 t2 則會包含剩下的元素

至於 merge 就是把兩個小 treap 合併成一個大 treap 的操作

淺談 split & merge

這邊稍微再講一下 split & merge

單純直接看函式會很難搞懂，建議先用很小的數據（像是只有 5 個 element 的 case）去做一次推導

split 的過程有點像是根據我們要的 index，如同閃電般割出一條線出來

那條線像是一條 path，有點摩西分紅海的 fu，會把一路上遇到的東西分成左右兩個 treap

詳細運作原理還是得靠自己看懂 code 才行

merge 同理，也是有一條像是縫合處的東西存在，合併時都會沿著那條 path 一路下去

搞懂這個 “path” 的概念有什麼用呢？這對於 reverse 這個操作有很大影響

因為是遞迴函式，所以全部 merge 完後，會根據那條 path 慢慢回推上來

這樣就能維護好待會要介紹的 lazy tag

lazy tag

這種方法目的是「等到需要更改時再進行更改」，藉此節省時間

什麼意思呢？假設我今天要把 [1, 100] 都 reverse，接著把 [5, 100] 再 reverse 一次

那假如我們每次都真的根據 operation 去操作，那肯定浪費一堆時間

像是 DSU 更改 group 的核心精神就是只更改它的頭頭一樣

lazy tag 的目的就有點像是說，假如我經過上述兩次 operation 後，只有要查詢 [1, 2] 的值

那我根本沒必要去 reverse [5, 100] 這段區間

lazy tag 一樣有兩個核心動作：pushdown (push) & pushup (pull)

pushdown 會寫在函式的最前面，目的是把該點的「資訊」（是否需要 reverse）傳遞給它的 child

pushup 則會寫在函式的最後面，目的是當操作進行完後，還必須把更改後的資訊對 parent 維護好
